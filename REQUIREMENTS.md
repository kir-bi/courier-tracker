# Courier Tracker — Требования

Версия: MVP v1.0
Дата: 2026-05-09
Статус: гипотеза → валидация → разработка

---

## 1. Контекст

**Проблема:** рестораны с собственной курьерской службой не видят, где находятся их курьеры в моменте. Клиенты звонят: «когда будет заказ?». Менеджеры отвечают «примерно через 40 минут» без реальных данных. История маршрутов не сохраняется — невозможно разобраться, почему курьер задержался.

**Решение:** простой инструмент трекинга — курьер делится геолокацией через мобильное приложение, менеджер видит всё на карте в веб-админке.

**Первый клиент для валидации:** Виталий (Kilogramm, Кайфуй — Полтава, Луцк, Сумы, Катовице).

**Не для:** агрегаторов (Яндекс.Еда, Glovo) с собственной логистикой.

---

## 2. Поверхности продукта

В MVP мы делаем **три и только три** поверхности:

| # | Поверхность       | Для кого     | Платформа                | Назначение                                          |
|---|-------------------|--------------|--------------------------|-----------------------------------------------------|
| 1 | **Лендинг**       | потенциальный клиент | веб (десктоп + мобайл) | продаёт продукт, собирает заявки                    |
| 2 | **Веб-админка**   | менеджер ресторана   | только веб (десктоп + планшет) | оперативный контроль курьеров и заказов |
| 3 | **Приложение курьера** | курьер              | iOS + Android (нативно или React Native) | минимальный интерфейс для смены и адресов |

**Чего НЕТ в MVP:**
- ❌ Мобильное приложение для менеджера. Менеджер сидит за компом или планшетом, в полевых условиях не работает.
- ❌ Любое клиентское приложение / страница live-трекинга для конечного клиента ресторана.
- ❌ Карта и навигация внутри приложения курьера.
- ❌ Любые публичные API и интеграции для третьих сторон (white-label, embed-виджеты и т.п.).

## 3. Роли пользователей

| Роль        | Описание                                      | Где работает         |
|-------------|-----------------------------------------------|----------------------|
| **Владелец** | Подписался на сервис, владеет аккаунтом ресторана. Может всё. | веб-админка |
| **Менеджер** | Сотрудник ресторана, ведёт смены, контролирует курьеров. | веб-админка |
| **Курьер**   | Развозит заказы, шарит локацию.                | мобильное приложение |

В MVP можно не разделять Владельца и Менеджера — одна роль с полным доступом. Декомпозицию ролей оставляем на V2+.

## 4. Лендинг

### 4.1 Назначение
Объяснить продукт, снять возражения, собрать заявку (email / телефон).

### 4.2 Структура (по дизайну в `design/wire-landing.jsx`)
1. **Hero** — заголовок «Знай где твой курьер. Всегда.», подзаголовок про ETA, CTA «Попробовать бесплатно», два мокапа: веб-админка + приложение курьера.
2. **Боль → Решение** — 3 карточки.
3. **Как это работает** — 3 шага. ⚠️ В MVP **третий шаг про "клиент получает ссылку" заменить** на формулировку без клиентской поверхности (например: «3. Сохраняешь историю — разбираешь маршруты по дням»).
4. **Скриншоты продукта** — два экрана: дашборд менеджера, приложение курьера. ⚠️ Третий «live-трекинг для клиента» **убрать**, оставить два.
5. **Тарифы** — 2-3 плана с ценой за курьера/месяц. ⚠️ Из фич плана «Бизнес» **убрать «API» и «white-label трекинг»** — это V2+.
6. **Финальный CTA** — заявка / email-форма.

### 4.3 Стиль
- Тёмно-синий hero (`#0d1b2a`), оранжевый акцент (`#ff6b1a`).
- Чередование тёмных и светлых секций.
- Мокапы устройств в hero.
- Вдохновение: PAJ GPS (paj-gps.es).

### 4.4 Что лендинг должен делать (формы)
- Кнопки CTA ведут на форму заявки: имя / название ресторана / телефон / город.
- Заявка падает в email владельца сервиса (или в Telegram). CRM не нужна.
- Никакой регистрации в один клик в MVP — только заявка → ручная активация.

## 5. Веб-админка для менеджера

### 5.1 Базовая навигация (по дизайну `wire-manager-web.jsx`)
Сайдбар:
- **Обзор** — KPI-плитки, ленты курьеров, последние заказы.
- **Заказы** — таблица всех заказов с фильтрами и поиском.
- **Карта** — основной рабочий экран: карта на весь экран, пины курьеров, плавающие панели.
- **Курьеры** — список курьеров, добавление, выдача доступа.
- **Точки** — управление точками доставки (см. 5.5). *Пункт меню добавить в дизайн.*
- **Отчёты** — история маршрутов и сводки за день/неделю.
- **Настройки** — профиль ресторана, биллинг, выход.

### 5.2 Карта (главный экран)
- **Live-карта** с маркерами всех активных курьеров.
- Цвет пина = статус: в пути (оранжевый), ждёт (синий), задержка (красный), офлайн (серый).
- Фильтры по статусу и по конкретному курьеру (чипы поверх карты).
- Тап / клик по пину → всплывающая мини-карточка: имя курьера, телефон, текущий заказ, ETA до точки, расстояние.
- Боковая панель справа — список активных заказов, синхронизирована с картой (выделение и скролл к выбранному).
- Поверх карты: счётчик курьеров онлайн (`4 / 6`), кнопка центрирования, зум.

### 5.3 Заказы — таблица (третий вариант макета)
Колонки: №, Адрес, Клиент, Курьер, Создан, Статус, ETA, действия.
Фильтры: статус (Все / В пути / Ждут / Задержка / Готово), курьер, период.
Поиск по №, адресу, имени клиента.

### 5.4 Создание / редактирование заказа
- Кнопка «+ Новый заказ» в шапке.
- Поля: адрес доставки (с автодополнением), имя клиента, телефон клиента, комментарий, назначение курьера (выбрать из списка свободных или «автоматом» = первому свободному).
- После создания заказ попадает в очередь курьера. Если курьер на смене — приложение тут же показывает новую точку.

### 5.5 Управление точками (Места доставки)
**Что это:** библиотека частых адресов, чтобы менеджер не вбивал каждый раз вручную.
- Список точек: адрес, координаты, метка (например, «офис клиента N»), частота заказов.
- Действия: добавить, отредактировать, удалить, отметить как «постоянную».
- При создании заказа — autocomplete сначала по этому списку, потом по геокодеру.

### 5.6 Управление курьерами
- Список: имя, телефон, статус (онлайн / офлайн / на смене), курьер активен / отключён.
- Добавление курьера: вводишь имя + телефон → система создаёт код (или магическую ссылку), курьер логинится в мобильном приложении.
- Возможность отключить / удалить курьера.

### 5.7 История маршрутов и отчёты (важно!)
Для каждого курьера за выбранный день:
- **Линия маршрута** на карте (треком GPS-точек, отрисованным polyline).
- **Тайм-лайн смены** под картой:
  - время старта смены, конца смены;
  - каждый заказ как сегмент: начало → точка доставки → завершение;
  - **сколько ехал** (движение между точками);
  - **сколько ждал на точке** (стоял на месте до нажатия «Я приехал»);
  - **сколько ждал до следующего заказа** (после завершения, до старта следующего).
- Сводка за день: выполнено заказов, общее расстояние, время в пути, время ожидания.
- Экспорт в CSV (опционально для MVP, можно отложить).

### 5.8 Real-time обновление
- Карта и список заказов обновляются автоматически (без перезагрузки страницы).
- Транспорт: WebSocket или Server-Sent Events. Polling только как fallback.
- Минимальная частота обновления позиции курьера на карте: каждые 5-10 секунд.

## 6. Мобильное приложение курьера

### 6.1 Главный принцип
**Минимум кнопок. Никакой карты. Никакой навигации.**
Курьер пользуется внешним приложением (Google Maps / Yandex Maps / 2GIS) — наше приложение работает в фоне и шарит локацию.

### 6.2 Аутентификация
- Логин по коду / магической ссылке от менеджера. Без регистрации.
- Один курьер привязан к одному ресторану.

### 6.3 Состояния (флоу — соответствует `wire-app.jsx` варианты CourierA + CourierB)

#### Состояние «Смена не начата» (CourierA)
- Большая центральная кнопка «**▶ НАЧАТЬ СМЕНУ**».
- Над кнопкой: «Сегодня · 6 заказов на смену», диапазон времени.
- Подсказка снизу: «Нажми, чтобы включить геолокацию».
- При нажатии → запрос разрешения на геолокацию (foreground + **background**), затем переход в активное состояние.

#### Состояние «Смена активна» (CourierB)
- Сверху оранжевая полоса со статусом: «● Смена активна · 1ч 23м» + ссылка «Завершить смену».
- Закреплённая карточка «**Сейчас**» — текущий заказ:
  - адрес крупно;
  - имя клиента + телефон;
  - три кнопки в ряд: **«Я приехал»** (главная, оранжевая, занимает большую часть ширины), 📞 (звонок клиенту), 🗺 (открыть адрес во внешней навигации).
- Под закреплённой карточкой — список «Дальше · N точек» с порядковым номером, адресом, именем клиента, временным слотом.
- При нажатии «Я приехал»:
  - текущий заказ помечается завершённым;
  - **система автоматически выдаёт следующий** — он становится текущим;
  - если заказы кончились → переход в состояние «Все заказы выполнены» (CTA «Завершить смену»).

#### Завершение смены
- По кнопке «Завершить смену» в шапке (с подтверждением).
- Останавливается шаринг локации.
- Показывается короткая сводка: сколько заказов, расстояние, время в работе.
- Возврат в состояние «Смена не начата» (на завтра).

### 6.4 Геолокация в фоне (критическое требование)
- Приложение должно **продолжать передавать координаты, когда свёрнуто** (background location).
- На iOS — режим `Always`, на Android — `ACCESS_BACKGROUND_LOCATION`.
- При потере связи — буферизовать точки локально и отправить пачкой при восстановлении сети.
- Включается только в активной смене. Завершил смену → шаринг прекращается.
- Уведомление в шторке: «Courier Tracker · Смена активна» (требование iOS / Android для background location).
- Энергоэффективность: периодичность апдейтов 5-10 сек, фильтр по «значимому перемещению» (significant location change) когда стоит на месте.

### 6.5 Чего в приложении курьера НЕ будет
- ❌ Карты (никакой `MapView` в MVP).
- ❌ Маршрута / прокладки пути.
- ❌ ETA расчётов.
- ❌ Чата с менеджером (только звонок клиенту).
- ❌ Истории и аналитики курьеру (это менеджеру в админке).
- ❌ Push-уведомлений о новых заказах в фоне (V2 — не критично, заказы появляются в списке когда курьер сворачивает в приложение).

## 8. Технические требования

### 8.1 Стек (гипотеза)
- **Мобильное приложение:** React Native (одна кодовая база) или нативно (если нужна более надёжная работа с background location). Решить после прототипа.
- **Веб-админка:** SPA — Next.js / React + Tailwind.
- **Карты:** Mapbox GL JS (тёмная тема, дешевле для real-time меток) или Google Maps. Решить после прикидки бюджета.
- **Бэкенд:** Node.js (или Go), PostgreSQL для данных + Redis для real-time каналов.
- **Real-time:** WebSocket (Socket.io / нативный WS).
- **Хостинг:** что-то простое — Railway / Fly.io / Hetzner.
- **Геокодирование адресов:** Mapbox Geocoding или Nominatim (бесплатно).

### 8.2 Нефункциональные требования
- **Latency:** позиция курьера на карте менеджера обновляется не реже чем раз в 10 сек.
- **Точность GPS:** хранить точку только если accuracy < 50 м.
- **Доступность:** uptime ≥ 99% (один регион, без HA).
- **Безопасность:** HTTPS везде; коды доступа для курьеров — одноразовые; данные одного ресторана не видны другим.
- **Хранение истории:** GPS-треки хранятся минимум 30 дней. Затем агрегируются (только сводки).
- **Локализация:** русский язык для интерфейса MVP. Английский — V2.

### 8.3 Дизайн-система
- Цвета: фон `#0f1923` (тёмная), `#fafaf7` (светлая), accent `#ff6b1a`, navy `#0d1b2a`.
- Шрифт: Inter / system-ui.
- Тёмная тема — основная. Светлая — для админки опционально (см. `wire-light.jsx`).
- Все вайрфреймы — в папке `design/`. Это источник правды для скоупа экранов в MVP.

## 9. Метрики успеха MVP

| Метрика | Цель |
|---------|------|
| Платящих ресторанов через 3 месяца после запуска | 3-5 |
| Курьеров на платформе через 3 месяца | 15-30 |
| Daily active managers (заходят в админку каждый день) | ≥ 80% от платящих |
| Среднее время от создания заказа до «Я приехал» | измеряем для бенчмарка |
| Сохранение между неделями (не отвалились) | ≥ 90% |
